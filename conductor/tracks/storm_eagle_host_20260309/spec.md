# Specification: Add Host `storm-eagle` (Mac Studio Asahi Linux)

## Overview
This track involves adding a new host, `storm-eagle`, to the NixOS configuration repository. `storm-eagle` is a Mac Studio with an M1 Max chip (64GB RAM, 2 M1 Max cores) running Asahi Linux. It will function as a headless server hosting a suite of Docker containers previously managed by an Ansible-based project (`ghostship-config`).

## Functional Requirements

### 1. Host Configuration
- Create a new host directory: `hosts/storm-eagle/`.
- Implement `hosts/storm-eagle/default.nix` with system-level configuration.
- Provide a placeholder `hosts/storm-eagle/hardware-configuration.nix` (to be updated later with actual host-generated config).
- Configure the system for headless operation.
- Ensure compatibility with Asahi Linux (Asahi-specific kernel, firmware, and bootloader settings).

### 2. Service Management (NixOS Modules)
- Implement a new module category: `modules/self-hosted/`.
- Create individual Nix modules for each OCI container service within `modules/self-hosted/`.
- A central `modules/self-hosted/default.nix` will manage the inclusion and configuration of these services.

### 3. User & Group Management
- Implement service-specific users for each containerized application.
- Use **statically mapped UID/GID** for all service users to ensure consistency across rebuilds.
- Create a shared group (e.g., `media-data`) for services that need shared access to the `/srv/apps/` directories.

### 4. Storage & Persistence
- Base directory for all persistent data and configuration: `/srv/apps/`.
- Dedicated database instances (containers) for each service requiring a database.
- Configure volumes and permissions to match the static UID/GID mappings.

### 5. Structured Configuration Merging Strategy (using `dasel`)
Implement a strategy that ensures Nix-managed settings are always reflected on the server while preserving app-managed state, using `dasel` as the primary manipulation tool.

#### Category A: Pure Nix-Managed (Read-Only)
- **Strategy**: Files are written to the Nix store and mounted read-only into the container.
- **Services**: `cloudflared`, `gluetun`, `metube`, `bentopdf`, `it-tools`, `zerobyte`, `manyfold`, `warracker`, `huntarr`, `flaresolverr`, `convertx`, `homepage`, `searxng`.

#### Category B: Structured Merging (Post-Initialization)
For applications that modify their own configuration, use a "Wait -> Inject -> Restart" strategy with `dasel`.
- **Strategy**: 
  1. Define the "Source of Truth" settings in Nix.
  2. Implement a one-shot `systemd` unit for each service that:
     - Depends on the OCI container starting.
     - Waits for the target configuration file to be generated by the application.
     - Uses `dasel` to **merge** the Nix-defined settings into the file.
     - Restarts the OCI container to apply the new settings.
  3. Ensure this runs on every `nixos-rebuild switch` if the Nix-defined settings have changed.
- **Services**:
  - `plex`: Merge settings into `Preferences.xml`.
  - `qbittorrent`: Merge settings into `qBittorrent.conf`.
  - `nzbget`: Merge settings into `nzbget.conf`.
  - `prowlarr`, `sonarr`, `radarr`, `bazarr`: Merge settings into `config.xml`.
  - `tautulli`: Merge settings into `config.ini`.
  - `homeassistant`: Merge settings into `configuration.yaml`.
  - `syncthing`: Merge settings into `config.xml`.
  - `muximux`, `plex-auto-languages`, `romm`, `booklore`, `recyclarr`: Use `dasel` for structured merging.

### 6. Secrets Management
- Use `sops-nix` to manage all environment variables and sensitive configuration data.
- Integrate secrets into the OCI container definitions (e.g., `environmentFiles`).

### 7. Networking
- Replicate the VPN routing logic (Gluetun) for `qbittorrent`, `nzbget`, and `searxng`.
- Ensure **all containers are accessible from each other** for seamless communication.
- Configure `cloudflared` to have access to all services in the stack.
- Use a dedicated Docker bridge network for internal service communication.

## Non-Functional Requirements
- **Reproducibility**: The entire host configuration must be reproducible via `nix flake`.
- **Security**: Minimize container privileges and use dedicated users where possible.
- **Maintainability**: Use a modular approach with the new `modules/self-hosted/` structure.

## Acceptance Criteria
- [ ] `storm-eagle` host is defined in `flake.nix`.
- [ ] `dasel` is added to common system packages.
- [ ] System successfully builds with a placeholder hardware configuration.
- [ ] New `modules/self-hosted/` module structure is established.
- [ ] OCI container definitions for ALL listed services are implemented as individual modules.
- [ ] Static UID/GID mappings for service users are defined.
- [ ] Secrets management via `sops-nix` is integrated.
- [ ] Data directory layout `/srv/apps/` is established in the configuration.
- [ ] Each database-dependent service has its own isolated database container.
- [ ] The `dasel`-based "Post-Initialization Merging" strategy is implemented and functional.
- [ ] Inter-container connectivity is verified in the configuration.

## Out of Scope
- Initial hardware configuration generation (to be performed on the host).
- Migrating actual data from the old server to the new one.
- Advanced performance tuning for Asahi Linux (beyond basic defaults).
