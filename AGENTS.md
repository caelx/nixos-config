# Project Agent Memories

This file serves as the primary memory and persistent fact store for AI agents within this project.

## Lessons Learned

- **CloakBrowser Origin Patch Point**: In `cloakhq/cloakbrowser-manager`, the clean site-wide place to neutralize browser `Origin` headers is the raw ASGI `AuthMiddleware` in `/app/backend/main.py`, not the individual VNC/CDP routes. A startup patch that inserts a helper before `class AuthMiddleware` and strips `scope["headers"]` for `http` and `websocket` requests preserves the native manager/WebSocket stack and avoids the fragile aiohttp proxy workaround.
- **Remote Activation Caveat**: On `chill-penguin`, if a remote non-interactive deployment reaches a built system path but the final activation step fails, apply the built generation directly as `root` with `/nix/store/<system>/bin/switch-to-configuration switch`.
- **PyLoad LSIO Startup Ownership**: `lscr.io/linuxserver/pyload-ng` changes `/downloads` ownership in `/etc/s6-overlay/s6-rc.d/init-pyload-config/run` via `lsiown abc:abc /downloads`, not via a safe static `fix-attrs` unit override. On `chill-penguin`, `user = "3000:3000"` also breaks LSIO startup because `/config` setup and later `s6-setuidgid abc` expect a root PID 1. Keep the container in the supported root-run mode with `PUID`/`PGID`, and patch that specific init script stanza by exact content match instead of mounting a `fix-attrs/down` file into `s6-rc.d`.
- **PyLoad Config Format**: `pyload.cfg` is its own typed config format, not KV/INI. `ghostship-config` must special-case the basename `pyload.cfg` and update existing `section.key` entries in-place (for example `download.max_downloads` and `webui.autologin`) instead of appending `section.key=value` lines at EOF, which PyLoad ignores and logs as parser errors.
- **PyLoad Healthcheck Path**: On `chill-penguin`, the PyLoad web process can be fully up while `/` and `/login` loop between `302 /login?next=...` and absolute redirects when `webui.autologin` is enabled. The container healthcheck should probe `http://127.0.0.1:8000/api`, which returns an anonymous `200 OK`, instead of `/`, or Podman will leave the container stuck in `starting`.

- **modDirVersion Alignment**: When using `prev.buildLinux` or similar kernel build functions in Nix, `modDirVersion` must exactly match the version string the kernel expects (often found in the top-level `Makefile`). A mismatch (e.g., `6.18.10-asahi` vs `6.18.10`) will cause a build failure late in the process during module installation.

- **yq-go INI Config Management**: For NixOS activation scripts managing INI config files:
  - Use `pkgs.yq-go` with `-p ini -o ini` flags for INI files
  - Use `pkgs.gnused` for sed commands in activation scripts (not coreutils sed)
  - Always reference tools with full path: `${pkgs.yq-go}/bin/yq`, `${pkgs.gnused}/bin/sed`
  - **INI-specific**: Delete old sections BEFORE adding new ones to preserve order
  - **Generic delete all sections except some**: `yq -i -p ini -o ini 'with_entries(select(.key == "section1" or .key == "section2"))' file.ini`
  - **Muximux-specific**:
    - yq adds backticks around `#` values - fix with: `sed -i 's/``\([^`]*\)``/"\1"/g'`
    - yq preserves PHP header when using `-i` flag
    - Homepage should be first service entry and only one with `default = "true"`
- **Homepage-specific**:
    - Use yq without `-p ini -o ini` flags (regular YAML parsing)
    - For large YAML configs, use `pkgs.writeText` to embed the YAML as a file, then `cp` to target
    - Use yq for partial updates (e.g., `.title = "value"`)

- **ISO Partitioning**: A custom ISO manually created from `iso-extract/` lacked a partition table (GPT/MBR), causing U-Boot's EFI boot manager to fail with `Cannot load any image`. The working ISO was built using the official `nixos-apple-silicon` repository with: `nix build .#installer-bootstrap -o installer -j4 -L`.
- **M1 Ultra Multi-Die Addressing**: M1 Ultra (t6002) peripherals on Die 1 have an 80GB offset (`0x2000000000`). The Asahi kernel (6.19.9) handles this correctly.
- **WiFi Firmware Compression**: The Asahi kernel (6.19.x) fails to load `brcmfmac` drivers if they are ZSTD compressed. Set `hardware.firmwareCompression = "none";` in the configuration.
- **Firmware Source Path**: The correct source for Apple firmware on NixOS is `/boot/asahi/`, containing `all_firmware.tar.gz` and `kernelcache*`.
- **Impure Rebuilds**: When `extractPeripheralFirmware` is enabled and pointing to `/boot/asahi`, `nixos-rebuild` REQUIRES the `--impure` flag to access files outside the flake.
- **Asahi Firmware Scope**: `/boot/asahi` is only relevant on `chill-penguin`; do not treat it as a repo-wide requirement. The peripheral-firmware module defaults `extractPeripheralFirmware = true`, so hosts without firmware must explicitly set it to `false` when `builtins.pathExists /boot/asahi` is false.
- **Gluetun Secret Ordering**: In this repo, `sops` installs secrets via activation scripts, not `sops-nix.service`. Do not add dependencies on `sops-nix.service`. Keep Gluetun's secret checks in `preStart` and wait for the actual files.
- **Podman Container Networking Dependencies**: Any container that uses `--network=container:gluetun` must explicitly declare `after = [ "podman-gluetun.service" ]` and `bindsTo = [ "podman-gluetun.service" ]`. Without that, the dependent container can survive a Gluetun replacement and block Podman from recreating the `gluetun` container.
- **Podman Healthcheck Failure Handling**: For Nix-managed Podman services in this repo, keep Podman's native healthcheck cadence (`--health-interval=30s`, `--health-timeout=10s`, `--health-retries=5`, `--health-start-period=1m`) and add `--health-on-failure=kill` so systemd's existing `Restart=on-failure` policy restarts the container. Do not rely on `podman healthcheck run` plus `--health-interval=disable` as a drop-in replacement: on `chill-penguin`, manual `podman healthcheck run` returned failure but left the container health state at `starting` with `FailingStreak=0`, so Podman's native retry/unhealthy tracking was not preserved.
- **Service Config Generation**: If a service config generator depends on `sops`-managed secret files, move it to `systemd.services.<unit>.preStart` instead of `system.activationScripts`. In this repo that is sufficient because `/run/secrets` is populated during activation before units are started.
- **Mutable Users Safety**: In early-stage hardware setups where WiFi might be unstable, set `users.mutableUsers = true;`. This prevents lockouts if `sops-nix` fails to decrypt the user password on boot.
- **Btrfs Layout**: A modern subvolume layout (`@`, `@home`, `@nix`, `@log`) with `compress=zstd` and `noatime` is preferred for large NVMe storage on Apple Silicon. When creating Btrfs filesystems on Apple Silicon, use 4K sectors (default) even if the CPU page size is 16K, as it is supported by kernels 6.x+ and ensures compatibility.
- **Native Nix Output Flags**: When building with `nix build`, prefer `-L` for logs or omit extra output suppression flags unless they are confirmed to work in the current environment.
- **Ghostship Secret Resolution**: `modules/common/scripts/ghostship-config.py` must accept multiple `--secrets-file` inputs because activation scripts often combine several service-local secret files. Merge them in order so later files can override earlier values.
- **Ghostship YAML Scalars**: Use `yaml:` patches in `ghostship-config.py` when YAML fields must be written with native scalar types (for example booleans and integers in `searxng` settings). `literal:` always starts as a string and is only safe when the file already has the correct target type.
- **Homepage Resources Widget Booleans**: Homepage's `resources` widget must receive native YAML booleans for `cpu`, `memory`, and `network`. If `ghostship-config.py` writes them as `literal:true`, Homepage's `/api/widgets` emits `"true"` strings and the dashboard's network bandwidth section falls back to `API Error`.
- **Homepage Resources Network Mount**: For Homepage on `chill-penguin`, mounting only `/sys/class/net` into the container is insufficient because the host interfaces inside that directory are symlinks into `/sys/devices/...`; without the matching `/sys/devices/platform` subtree, `end0/statistics/*` is missing. The container's own default route is `eth0`, while the host traffic you want is on `end0`, so keep `network: end0` and mount `/sys/class/net` plus `/sys/devices/platform`, not the entire host `/sys`.
- **Homepage `/sys` Scope Matters**: Do not bind the entire host `/sys` into Homepage just to fix network stats. On `chill-penguin`, that exposed `/sys/firmware/efi/efivars` as a mountpoint whose target path is absent in the container, `df -P -T` emitted `No such file or directory`, and Homepage's `systeminformation.fsSize()` returned `[]`, breaking the disk widget. Keep `network: end0`, but mount only `/sys/class/net` plus the needed `/sys/devices/platform` subtree so the NIC symlinks resolve without polluting disk detection.
- **Ghostship YAML Paths**: `ghostship-config.py` YAML paths must handle dots inside bracket selectors (for example `[Infrastructure].[Llama.cpp]`), bracketed numeric list indices (for example `custom_formats[0].trash_ids[0]`), and bracket selectors against YAML mappings with dotted keys (for example `plugins[searx.plugins.calculator.SXNGPlugin].active`). Homepage, Recyclarr, and SearXNG all depend on those exact path forms.
- **Ghostship Homepage Group Creation**: When `ghostship-config.py` creates missing Homepage-style YAML groups (for example `[Utilities].[OmniTools].icon`), the new top-level group container must be a list, not a mapping. If the next path segment is another bare bracket selector, create `[]`; otherwise create `{}`.
- **Ghostship Package Lint Gate**: `pkgs.writers.writePython3Bin` is lint-gated here. `ghostship-config.py` must satisfy the package style checks too, including 79-column line length, blank-line spacing, and no trailing whitespace, before claiming the package builds.
- **Python Style Fix Workflow**: When a Python build fails on formatting-only issues in this repo, prefer an autopep8-capable formatter with line-wrapping support to clear `E501` and related style violations quickly before rerunning package builds.
- **qBittorrent Config Format**: `qBittorrent.conf` is an INI file despite the `.conf` extension. `ghostship-config.py` must special-case that basename as INI, and qBittorrent WebUI settings belong under `[Preferences]` as `WebUI\\...` keys, not legacy `WebUI.*` KV lines.
- **Host Port Exposure**: Do not expose container ports on the host system, except for Plex. All other services should use internal networking or reverse proxies.
- **SearXNG Access Denied (VPN)**: When running behind Gluetun, certain engines permanently block the VPN IP with "Access Denied" errors: `karmasearch`, `vimeo`, and `genius`. Keep these disabled in `settings.yml` to avoid cluttering results with errors.
- **VueTorrent LSIO Mod Path**: On `chill-penguin`, qBittorrent 5.1.4 persistently rewrites the alternative WebUI path to `/vuetorrent-ui/public`, and even a fresh manual extraction of the current upstream `vuetorrent.zip` still returns `Unacceptable file type, only regular file is allowed.`. Use the supported LinuxServer VueTorrent mod (`DOCKER_MODS=ghcr.io/vuetorrent/vuetorrent-lsio-mod:latest`) and configure `WebUI\\RootFolder=/vuetorrent` instead of bind-mounting a hand-managed archive.
- **VueTorrent Docker Mods Require Root Entrypoint**: The LinuxServer qBittorrent image disables Docker mods when the container is forced to run as a non-root user with `-u 3000:3000`. For VueTorrent on this host, keep `PUID=3000` and `PGID=3000` but do not set `virtualisation.oci-containers.containers.vuetorrent.user`, or the LSIO mod will not apply.
- **Root Podman Verification**: On `chill-penguin`, the Nix-managed Podman services use the root Podman store. A regular user `podman ps` can be empty even while `podman-*.service` is active. Verify live containers and `podman exec` checks from the `agentroot` tmux session or another root shell.
- **Gluetun Live Self-Heal**: On `chill-penguin`, forcing `tun0` down inside the running Gluetun container did not restart the container or `podman-gluetun.service`, but Gluetun's internal VPN healthcheck detected the routing failure about 30 seconds later, restarted the VPN process in-place, and restored outbound connectivity with a new public IP. `gluetun-network-monitor` then logged the IP change and restarted `podman-vuetorrent.service` plus `podman-nzbget.service`; `podman-searxng.service` stayed up because it is not part of that IP-change restart list.
- **VueTorrent Port Sync Monitor**: On `chill-penguin`, `gluetun-network-monitor` also repairs qBittorrent's listen port when it drifts from Gluetun's forwarded port. A live test forced qBittorrent from `47736` to `54321`; within the next monitor cycle, it logged `VueTorrent port mismatch (Current: 54321, Forwarded: 47736). Updating...` and restored `listen_port` to `47736`.
- **Cloudflared Secret Names**: Keep Cloudflared's credentials split by purpose in `cloudflared-secrets`: `CLOUDFLARED_TUNNEL_TOKEN` is for the `cloudflared` container itself, while Homepage's Cloudflared widget needs `CLOUDFLARED_ACCOUNT_ID`, `CLOUDFLARED_TUNNEL_ID`, and `CLOUDFLARED_API_TOKEN`. Reusing the tunnel token for the widget returns Cloudflare `400` auth failures and leaves `connections` null.
- **Cloudflared Secret Mirror**: When editing `secrets.dec.yaml`, preserve the existing service blocks and append Cloudflare's `CLOUDFLARED_*` values into the `cloudflared-secrets` block. Do not replace the whole file with only the new block.
- **Cloudflared Tunnel Startup**: The `cloudflared` container must get a runtime `TUNNEL_TOKEN` env file and run with `cmd = [ "tunnel" "run" ]`. Passing a shell wrapper through `cmd` just becomes an extra `cloudflared` argument and still exits with "You did not specify any valid additional argument".
- **Cloudflared Image Debugging**: The `docker.io/cloudflare/cloudflared` image in this repo is shell-less, so connectivity checks against its network namespace should use a helper container with `--network container:cloudflared` rather than `podman exec ... sh`.
- **RomM iframe blocker**: `romm.ghostship.io` can return a healthy `200` from the origin container, but the public hostname is still protected by Cloudflare Access. Unauthenticated requests are redirected to the Access login page, and that login page sends `X-Frame-Options: DENY` plus `Content-Security-Policy: frame-ancestors 'none'`, so an iframe will fail before RomM renders unless the hostname is bypassed/unprotected or the browser already has a valid Access session.
- **RomM iframe follow-up**: If the browser shows `STATUS_BREAKPOINT` plus repeated sandbox warnings, check the parent portal too. In this repo the failing subrequest can be `https://apps.ghostship.io/images/favicon/manifest.json`, which Cloudflare Access redirects to the login page and then CORS-blocks. That means the embed breakage may be in the portal's own protected assets rather than in RomM.
- **Muximux manifest was incidental**: Stripping the `images/favicon/manifest.json` link from Muximux did not stop the `STATUS_BREAKPOINT` iframe crash. Treat the manifest redirect/CORS issue as a secondary symptom, not the root cause.
- **RomM iframe embed surface**: The live RomM shell emits two manifest links (`/site.webmanifest` and `/manifest.webmanifest`) from the generated HTML, and the bundle includes autofocus behavior in Vuetify text fields. The bundle does not appear to register a service worker. That makes RomM's own startup path a more plausible iframe-specific trigger than the earlier Muximux manifest lead.
- **RomM manifest suppression did not help**: Adding `manifest-src 'none'` only for iframe requests did not stop the `STATUS_BREAKPOINT` crash. The next hypothesis should focus on autofocus or another startup side effect, not the manifest fetch path.
- **RomM live-test pitfall**: Repo edits to `modules/self-hosted/romm.nix` do not affect the running RomM container until the host is rebuilt. When debugging iframe behavior, verify the live `/etc/nginx/templates/default.conf.template` or `/var/www/html/index.html` inside the container before treating a repo-only patch as tested.
- **RomM ships a service worker**: The live container serves `/sw.js`, `/workbox-*.js`, and both `/manifest.webmanifest` plus `/site.webmanifest`. The PWA surface is real even though the main bundle did not show an obvious `navigator.serviceWorker` string during quick greps.
- **RomM iframe main-document headers are not the blocker**: A live browser capture of `https://romm.ghostship.io/` inside the iframe showed `Content-Security-Policy: frame-ancestors https://apps.ghostship.io 'self';`, `Cross-Origin-Embedder-Policy: unsafe-none`, and `Cross-Origin-Opener-Policy: unsafe-none`, with no visible `X-Frame-Options`. When the iframe still crashes, look past document framing headers.
- **RomM cookie rewrite did not stop the iframe crash**: The running container's backend currently configures `RedisSessionMiddleware` with `same_site="strict"` when OIDC is disabled and `https_only=False` in `/backend/main.py`, and a live nginx test successfully rewrote `romm_session` / `romm_csrftoken` to `SameSite=None; Secure`. The iframe still crashed afterward, so cookie attributes are not the sole cause and should not remain the primary theory once mount/render evidence is available.
- **Cloudflare response expressions can read response headers**: Response Header Transform Rules can reference `http.response.headers["set-cookie"][0]` in dynamic expressions. The real caveat is specific to `set-cookie`: `Set dynamic` overwrites all existing `Set-Cookie` headers, while `Add dynamic` appends another one. That means Transform Rules are only a safe cookie-rewrite tool when each matching response emits exactly one relevant cookie header, otherwise use Snippets or fix the origin.
- **RomM/apps cookie scope correction**: `apps.ghostship.io` and `romm.ghostship.io` are different origins but the same site (`https` + `ghostship.io`). A host-only cookie set by RomM stays scoped to `romm.ghostship.io`; it does not become an `apps.ghostship.io` cookie just because RomM is framed there. This also means `SameSite=Strict` alone is not enough to explain the iframe crash between these two subdomains.
- **RomM preStart template bug**: The generated `podman-romm-pre-start` script in this repo writes the Nginx template with shell-expanded `${ROMM_PORT}`, `${IPV6_LISTEN}`, and `${ROMM_BASE_PATH}`. Because systemd does not define those shell vars, the live template becomes `listen ;` and `alias "/library/";`, which makes nginx crash on container restart. Fix the Nix heredoc/escaping before treating any RomM restart failure as an app bug.
- **RomM iframe crash is in app mount/render, not startup fetches**: A no-boot test page that loads the real bundle without calling `RG()` does not crash. A split test where `yG(e); await CG(); e.use(Ll);` runs without `mount()` also does not crash. A test that does `yG(e); e.use(Ll); e.mount("#app");` without `CG()` still crashes. Treat the failure as a frontend render/mount problem in the real RomM component tree or router path, not a network-startup or cookie issue.
- **RomM iframe view-transition hook is not the sole trigger**: The live router installs `Ll.beforeResolve(async()=>{ await LU().captured })`, and `LU()` calls `document.startViewTransition()` when available. An iframe-only test bundle that short-circuits that path with `if(window.top!==window.self)return n;` still crashes. Keep router/render logic in scope, but do not assume Chrome's View Transition API alone explains the iframe crash.
- **Cloudflare SSH MAC Mismatch**: On `chill-penguin`, `sshd -T` currently permits only `*-etm` and `umac-128-etm` MACs. Cloudflare SSH/browser sessions were observed offering only `hmac-sha2-256,hmac-sha2-512`, which causes `no matching MAC found` before auth even starts. When debugging Cloudflare SSH failures, check the server MAC policy before chasing key/auth issues.
- **Cloudflare SSH Compatibility Fix**: A host-only `services.openssh.macs` override on `hosts/chill-penguin/default.nix` is the clean way to make Cloudflare SSH compatible on this machine. It should include the plain `hmac-sha2-256` and `hmac-sha2-512` MACs in addition to the `*-etm`/`umac-128-etm` variants. Avoid adding a separate `settings.MACs` entry, because the NixOS SSH module already maps `services.openssh.macs` onto `settings.Macs` and will reject duplicate keys.
- **RomM root component is not the iframe crash trigger**: Both a stub mount (`yG(e)` plus a trivial component) and a `root-no-router` mount of the compiled `RomM` root component complete without crashing in the iframe. The remaining suspect surface is the first real route component render after the router is installed.
- **RomM setup route render is enough to crash Chrome in the iframe**: A fresh asset URL that merely imports the original `Setup-rt7B6UJc.js` module does not crash, but a fresh direct-mount bootstrap that renders that original `Setup` component with RomM's normal plugin stack does crash. The remaining suspect surface is the `Setup` component's render tree or one of its immediate child components, not module evaluation or the router alone.
- **RomM Setup shell is not the trigger; the real pane content is**: An iframe test that renders only the Setup stepper header works, another that renders the stepper with empty window panes also works, and simplified Setup variants with `We()` retained still work. Treat the remaining suspect surface as the actual rendered content inside Setup's step panes, not the stepper/window framework itself.
- **RomM Setup pane 3 is the iframe crash trigger**: Direct-mount variants that keep only one real Setup pane showed panes 1 and 2 render safely, while pane 3 alone still crashes. Focus the next split on the metadata-source row components in step 3 (`VListItem`, `VAvatar`, `VImg`, and the append checkmark span), not on the library-detection or admin-user panes.
- **A single original pane-3 row can trigger the iframe crash**: Reducing the metadata-source pane to just the first original row (`oe.value.slice(0,1)`) still crashes in the iframe. The crash does not require multiple rows; one full `VListItem` with the original prepend and append slots is enough.
- **Manual row recreation works; the crash follows `VListItem` composition**: A pane-3 variant that recreates the same avatar, text, and status glyphs with plain `div` rows does not crash, while the original single-row `VListItem` version does. That means the remaining suspect surface is Vuetify `VListItem` with both slots active, not the underlying image assets or glyphs by themselves.
- **The RomM row crash needs the full original triple combination**: A single-row `VListItem` with both slots and trivial slot payload (`vlimin`) renders safely, and a single-row `VListItem` with the original prepend image/avatar plus original append status glyph but no subtitle (`orignosub`) also renders safely. The remaining failing combination is the original subtitle together with the original prepend (`VAvatar` + `VImg`) and original append status slot on the same `VListItem`.
- **Fresh routed assets still crash**: A cache-busted routed `/setup` page using fresh filenames (`index-C1YMu947-livefix.js` and `Setup-livefix.js`) still crashes in the iframe even after replacing the metadata-source rows with the safe plain-row implementation. Do not assume the remaining failure is just stale caching of `Setup-rt7B6UJc.js`; there is at least one additional trigger in the real routed/auth startup path.
- **Two separate iframe triggers exist now**: A fresh single-row retest confirms the original pane-3 metadata row still crashes while the manual-row equivalent still works, and a fresh direct-mount page using the patched full `Setup-livefix.js` works. Treat the metadata-row bug as real and independently fixed in the direct-mount case; the remaining routed `/setup` crash comes from another trigger in the real router/auth path.
- **The second trigger is specifically in the routed app path**: A fresh routed `/setup` test with `Setup-livefix.js` and `CG()` removed still crashes, while a fresh `Auth` shell that mounts `Setup-livefix.js` directly without the router works. That rules out `CG()` and the auth shell itself. The remaining suspect surface is the real router path or the `RomM` root wrapper around `router-view`.
- **`authrouter` debug pages are not router-only isolates**: The `romm-authrouter-livefix-test.html` and `romm-authrouter-livefix-novt-test.html` experiments mounted `Auth` as the app root while the `/setup` route also renders `Auth`, which double-nests the auth shell. Treat their crashes as mixed router-plus-duplicate-Auth results, not clean proof that the router alone is the remaining trigger.
- **Clean minimal router with patched Setup works**: Two fresh tests using a custom single-route `/setup` router and `Setup-livefix.js` both work in the iframe: one mounts the original `RomM` root `E5`, and the other mounts a trivial `router-view` root. That rules out the `RomM` root wrapper, generic `vue-router` rendering, and the patched `Setup` route itself. The remaining trigger is in the real full router configuration, most likely its guards or other route-table-specific behavior.
- **Real RomM `/setup` redirects immediately**: The live heartbeat from inside the RomM container reports `SYSTEM.SHOW_SETUP_WIZARD = false`. In the real app, navigating to `/setup` does not render the Setup wizard; the guard redirects to `login` when unauthenticated or `home` when authenticated. Treat earlier “routed `/setup` crash” conclusions as ambiguous until the actual post-guard target route is confirmed.
- **Chunk-based iframe tests were contaminated by Vite's main-bundle side effect**: Route chunks like `Auth-DEfpmLbn.js`, `Login-CnAfCT08.js`, `Main-CyWiqIVJ.js`, and `Setup-rt7B6UJc.js` import `./index-C1YMu947.js` for shared helpers. That main bundle executes `RG()` at module top level. Any ad-hoc test that imports one of those chunks without first replacing that dependency boots the original app in parallel and is not a clean isolation. Use a patched no-boot runtime copy for future chunk-level tests.
- **RomM login iframe crash is triggered by the real router `beforeResolve` hook**: On the live host-published RomM port, a stock iframe of `/login?next=/` wedges `agent-browser` so even `get title` times out, but an otherwise identical live bundle variant that only changes `Ll.beforeResolve(async()=>{await LU().captured})` to a no-op stays responsive. The current live login-path iframe crash is therefore tied to the router's `LU().captured` / `document.startViewTransition()` path, not to framing headers, cookies, `CG()`, or the login component by itself.
- **RomM iframe preload shim was insufficient**: Injecting an iframe-only `index.html` script that tried to undefine `document.startViewTransition` before the stock bundle loaded did not stop the live `/login?next=/` iframe wedge. Treat the startup hook itself as the reliable patch point; a preload shim is not enough here.
- **Live RomM noresolve bundle swap preserves basic login behavior**: Replacing the running `/var/www/html/assets/index-C1YMu947.js` with the previously verified `beforeResolve(async()=>{})` variant made the stock iframe probe stay responsive (`get title` returned immediately) while the top-level `http://192.168.200.82:18080/login?next=/` page still loaded with title `Login`. For live testing, that bundle swap is the current working mitigation.
- **Muximux itself is not the RomM iframe crash trigger**: A disposable live `muximux-romm-test` container exposed on `http://192.168.200.82:18081/` with its RomM tile pointed directly at `http://192.168.200.82:18080/` loaded RomM successfully in `agent-browser`. The iframe rendered the login form and the page title stayed responsive (`RomM - ghostship.io`). If the normal Muximux embed still crashes, the remaining variable is the public `https://romm.ghostship.io` path (Cloudflare Access/public-origin state), not Muximux's embed mechanics.
- **RomM live-debug cleanup must exec as root**: Ad hoc debug assets written into the RomM container during iframe investigation ended up root-owned under `/var/www/html`. A `podman exec` hook running as the container's normal `3000:3000` user can patch the main bundle but cannot remove those files. Any persistent `postStart` cleanup/patch hook for RomM should use `podman exec -u 0`.
- **RomM postStart parser must avoid nested shell-quoted sed**: The first persistent `podman-romm` `postStart` implementation used a heavily escaped `sed -n 's@...@...@p'` fragment inside `writeShellScript` and `podman exec ... sh -lc`. On deployment it produced `sed: unsupported command "` inside the container and blocked `podman-romm.service` startup under `sysinit-reactivation.target`. Use a direct `python` parser/rewrite inside the container instead of layered shell quoting for this hook.
